# 番茄章节发布工具桌面化设计

## 1. 背景

当前番茄章节发布工具已经使用 React、Express 和 Playwright 跑通核心流程：导入本地 Markdown 章节、生成发布计划、进入番茄章节管理、创建章节、填写章节信息并设置定时发布。

当前使用方式仍需要分别启动前端和后端，再通过浏览器打开工具页面。本次桌面化的目标是让用户通过一个安装完成的桌面应用直接使用，同时保留现有网页开发模式，方便后续继续增加功能。

## 2. 目标

第一阶段桌面版需要达到以下目标：

- 同时支持 Windows 和 macOS。
- 双击应用即可启动，不显示命令行窗口，不要求用户手动运行前后端。
- 桌面应用主窗口负责导入、排期、状态和设置。
- 发布任务继续使用独立 Chrome 窗口操作番茄作者后台。
- 最大程度复用现有 React、Express 和 Playwright 代码。
- 保留现有网页开发模式，桌面版和网页开发版共享同一套核心代码。
- 小说正文、配置、登录资料和发布记录全部保存在本地。
- 通过 GitHub 自动生成 Windows 和 macOS 安装包。
- 为多账号、自动更新、代码签名和商业化保留扩展边界，但第一阶段不实现这些功能。

## 3. 第一阶段不做的功能

- 多番茄账号管理。
- 小说与账号绑定界面。
- 应用内自动更新。
- 内置专用浏览器。
- Windows 和 macOS 代码签名及 macOS 公证。
- 云端同步小说正文或发布记录。
- 商业授权、付费、用户登录和设备管理。
- 浏览器插件版本。

## 4. 方案选择

采用 Electron 桌面应用，并在应用内部启动现有 Express 服务。

不采用浏览器插件，原因是插件不适合直接读取任意本地小说文件、运行 Playwright、管理独立 Chrome 和维护本地发布记录。插件最终仍需要额外安装本地程序，使用和维护成本更高。

不采用 Tauri，原因是现有 Node 和 Playwright 服务需要作为额外程序打包，跨平台构建和调试复杂度高，安装包体积优势不足以抵消改造成本。

暂不把所有 Express 接口改造成 Electron 内部通信，以免大范围重构已经跑通的核心发布流程。

## 5. 总体架构

桌面版分为四个边界清晰的部分：

### 5.1 React 界面

负责以下内容：

- 选择或输入小说文件夹。
- 导入和展示章节。
- 设置章节范围和每日发布时间。
- 生成、检查和调整发布计划。
- 启动、暂停、继续和停止发布任务。
- 展示发布状态、日志和错误提示。
- 展示数据保存位置、版本信息和卸载入口。

### 5.2 Express 本地服务

继续负责以下内容：

- 解析小说文件夹和 Markdown 章节。
- 生成发布计划。
- 读写 `.fanqie-publish.json`。
- 管理发布任务状态。
- 调用番茄自动化模块。

桌面应用启动时自动选择一个空闲的本机端口启动服务，不再固定依赖开发端口。服务只监听 `127.0.0.1`。

打包模式下，Express 同时提供构建后的 React 静态页面，主窗口从同一个本地地址加载界面。开发模式仍由 Vite 提供页面和热更新。

每次启动应用时生成临时访问令牌。桌面界面调用本地 API 时必须携带该令牌，避免其他网页误调用发布接口。

### 5.3 Electron 桌面外壳

负责以下内容：

- 应用单实例运行。
- 创建和管理桌面主窗口。
- 在后台启动和停止 Express 服务。
- 提供系统小说文件夹选择器。
- 管理应用数据目录。
- 检测本机 Chrome。
- 处理最小化、关闭确认和异常退出。
- 显示版本信息和打开新版下载页面。
- 执行数据清理和卸载流程。

Electron 的渲染进程不直接获得完整 Node 权限。需要使用的桌面能力通过受限的预加载接口提供。

### 5.4 番茄自动化模块

继续使用现有 Playwright 发布逻辑，负责：

- 启动本机已安装的 Chrome。
- 使用应用统一管理的浏览器登录资料。
- 进入番茄作者后台和目标书籍。
- 创建章节、填写章节信息、执行内容检测并设置定时发布。
- 验证发布结果并记录状态。

第一阶段使用本机 Chrome。以后商业版可以增加内置专用浏览器，而不改变上层发布任务接口。

## 6. 启动和退出流程

### 6.1 启动流程

1. 用户双击桌面应用。
2. Electron 检查是否已有应用实例。
3. 如果已有实例，则激活已有窗口，不重复启动服务。
4. 应用检查数据目录和配置版本，必要时执行配置迁移。
5. 应用检测本机 Chrome。
6. 应用选择空闲端口，生成临时令牌并启动 Express 服务。
7. 内部健康检查通过后创建主窗口。
8. 主窗口加载 React 界面。
9. 如果上次任务异常中断，界面提示用户检查未完成任务。

找不到 Chrome 时，应用仍可进入主界面查看数据和设置，但不能开始发布，并显示明确的 Chrome 安装或定位说明。

### 6.2 运行行为

- 最小化主窗口不影响发布任务。
- 开始发布后继续打开独立 Chrome 窗口。
- 用户可以使用其他应用，但不应手动操作发布专用 Chrome。
- 重复双击桌面图标只会切回当前窗口。

### 6.3 退出流程

- 空闲状态关闭主窗口时直接退出。
- 发布进行中关闭主窗口时，必须确认是否停止任务。
- 用户取消退出时，任务继续运行。
- 用户确认退出时，停止当前任务、关闭发布专用 Chrome、停止 Express 服务，然后退出应用。
- 应用异常退出时保留最后任务状态，下次启动提示人工确认，不能盲目重复发布。

## 7. 本地数据设计

### 7.1 应用数据目录

应用使用操作系统标准目录：

- Windows：当前用户的 `AppData` 应用目录。
- macOS：当前用户的 `Library/Application Support` 应用目录。

应用目录保存：

- 应用配置和配置版本。
- 最近使用的小说文件夹。
- 默认番茄账号定义。
- Chrome 登录资料。
- 运行日志和诊断信息。
- 已生成文件清单。
- 上一次发布任务的恢复信息。

覆盖安装新版本时不得删除应用数据。

### 7.2 小说文件夹数据

小说文件夹继续保存：

- 用户原始 Markdown 文件。
- 工具生成的 `.fanqie-publish.json` 发布记录。

工具不复制、不上传、不修改原始 Markdown 正文。

### 7.3 已生成文件清单

应用维护一份清单，记录它在哪些小说文件夹中生成过文件。清单至少包含：

- 文件完整路径。
- 文件类型。
- 对应小说文件夹。
- 创建或最后更新时间。

该清单用于数据展示、清理和卸载。文件被用户移动或已经不存在时，清理结果中应显示“未找到”，不能把它当成成功删除。

## 8. 账号和登录资料

第一阶段只提供一个默认番茄账号，内部账号编号为 `default`。

所有小说共用该账号的 Chrome 登录资料。桌面版首次使用时需要重新登录一次番茄，之后复用登录状态。

账号和浏览器资料的接口不能写死为单实例。后续多账号版本可以增加：

- 账号列表。
- 每个账号独立的 Chrome 登录目录。
- 小说与账号的绑定关系。
- 导入或发布前的账号选择。

现有章节发布流程只接收账号上下文，不负责账号管理，从而避免未来增加多账号时重写发布自动化。

## 9. 桌面界面调整

保留现有主页面结构，增加以下能力：

- “选择小说文件夹”系统按钮。
- 手动路径输入框继续保留。
- 最近使用的小说列表。
- 设置入口。
- Chrome 检测和登录状态。
- 当前应用版本和新版下载入口。
- 运行日志和诊断包导出入口。
- 数据与存储页面。
- 清除数据和卸载入口。

### 9.1 数据与存储页面

页面必须显示：

- 应用数据目录。
- Chrome 登录资料目录。
- 日志目录。
- 已生成文件清单。
- 各类数据占用空间。

每个可访问目录提供“打开文件夹”按钮。路径必须以用户可以复制的完整地址显示。

## 10. 数据清理和卸载

### 10.1 清理范围

卸载时分为两类数据：

1. 应用数据：配置、账号登录资料、缓存、日志、最近记录和恢复信息。卸载时必须删除。
2. 小说文件夹数据：`.fanqie-publish.json` 和其他工具生成文件。默认勾选删除，但允许用户取消勾选保留。

卸载前必须展示准备删除的完整路径并二次确认。界面需要明确提醒：删除发布记录后，以后可能无法自动识别已经提交过的章节。

### 10.2 Windows

Windows 提供标准卸载程序。无论从应用内还是系统“已安装的应用”进入，都需要执行清理流程。

### 10.3 macOS

macOS 在应用内提供“卸载并清理”入口。完成数据选择和清理后，将应用移入废纸篓。

### 10.4 清理结果

- 成功删除的路径显示成功状态。
- 用户选择保留的文件显示保留状态。
- 不存在的路径显示未找到。
- 无权限或删除失败的路径显示失败原因和处理建议。
- 不能在存在失败项时提示“已全部清理”。

## 11. 错误处理和诊断

界面使用用户可以理解的中文提示，不直接展示冗长的技术堆栈。

需要处理的桌面异常包括：

- 未安装或找不到 Chrome。
- 内部服务端口冲突或启动失败。
- 本地 API 健康检查失败。
- 发布专用 Chrome 被手动关闭。
- 番茄登录失效。
- 番茄页面结构变化。
- 应用异常退出。
- 发布结果无法明确确认。
- 数据清理或卸载失败。

发布结果不明确时，当前章节状态应标记为“待人工确认”，并立即暂停。不能直接重发，避免重复发布。

### 11.1 日志

- 日志只保存在本地。
- 日志按日期和大小轮转，自动清理旧日志。
- 设置页显示日志目录并提供打开按钮。
- 用户可以导出诊断包。
- 诊断包默认不得包含小说正文、Chrome Cookie、密码或完整认证信息。
- 应用不自动上传日志或诊断包。

## 12. 构建和版本发布

使用 GitHub Actions 分别在 Windows 和 macOS 环境构建：

- Windows 生成 `Setup.exe`。
- macOS 生成 `.dmg`。

发布流程应在创建版本标签后自动运行测试、构建并上传安装包到 GitHub Release。

第一阶段安装包不签名：

- Windows 可能显示“未知发布者”。
- macOS 可能提示无法验证开发者。
- 使用文档需要提供首次允许安装和打开的说明。

第一阶段不实现应用内自动更新。设置页显示当前版本号和 GitHub 新版下载入口。用户下载新版后覆盖安装，应用数据必须保持不变。

后续商业版可以在相同构建流程中增加：

- Windows 代码签名。
- Apple Developer ID 签名。
- macOS 公证。
- 应用内版本检查和自动更新。

## 13. 开发模式

必须保留现有网页开发方式：

- Vite 提供前端热更新。
- Express 独立运行后端服务。
- 桌面开发模式可以加载 Vite 页面并连接开发服务。
- 打包模式加载构建后的静态页面并自动启动内部服务。

桌面版和网页开发版共享章节解析、排期、发布记录和番茄自动化代码，不复制两套业务逻辑。

## 14. 测试设计

### 14.1 现有功能测试

继续运行并扩展章节解析、排期、发布记录、发布控制和自动化选择器测试。

### 14.2 桌面集成测试

验证：

- 单实例启动。
- 内部服务自动启动和停止。
- 动态端口和临时令牌。
- 系统文件夹选择器接口。
- Chrome 检测。
- 最小化和关闭确认。
- 数据目录展示。
- 覆盖安装后的数据保留。
- 数据清理和卸载选择。

### 14.3 模拟发布测试

使用本地模拟番茄页面，完整测试：

- 进入章节管理。
- 创建新章节。
- 填写章节号、标题和正文。
- 处理错别字提示。
- 选择内容检测。
- 设置日期和时间。
- 确认发布。
- 写入本地发布记录。

模拟测试不得修改真实番茄账号。

### 14.4 安装包测试

GitHub 在 Windows 和 macOS 分别构建并执行启动检查。安装包必须能安装、启动并打开主窗口。

### 14.5 真实验收

正式声明桌面版完成前，需要验证：

- Windows 安装、启动、选择小说文件夹和启动专用 Chrome。
- macOS 安装、启动、选择小说文件夹和启动专用 Chrome。
- 至少一次经过用户明确授权的真实单章定时发布。
- 番茄后台章节状态为“待发布”。
- 番茄发布时间与本地计划一致。
- `.fanqie-publish.json` 记录正确。
- 覆盖安装后数据仍然存在。
- 卸载时应用数据被删除，小说文件夹记录按用户选择删除或保留。

只有上述真实验收通过后，才能声明桌面版功能完成。自动测试和构建通过但真实验收未执行时，必须明确说明“构建通过，尚未完成真实发布验收”。

## 15. 成功标准

第一阶段成功标准：

- 用户不需要安装 Node.js 或手动运行任何命令。
- 双击应用即可打开番茄章节发布工具。
- 已有核心发布流程在桌面版中保持可用。
- Windows 和 macOS 安装包均由 GitHub 自动生成。
- 应用数据位置在界面中清楚可见并可以打开。
- 覆盖安装不会丢失登录状态、配置和发布记录。
- 卸载流程可以清理应用数据，并让用户决定是否删除小说文件夹中的工具记录。
- 网页开发模式继续可用，后续功能只维护一套核心代码。
- 第一阶段的结构允许以后增加多账号、自动更新、内置浏览器和商业授权。
