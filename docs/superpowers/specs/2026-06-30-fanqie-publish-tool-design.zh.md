# 番茄章节发布工具设计

## 目标

做一个本地网页工具，用来导入一本小说的 Markdown 章节文件，生成定时发布计划，并通过独立的 Playwright 浏览器自动提交到番茄作者后台。

第一版只聚焦“一本书、一批章节、一次发布计划”。核心目标是把重复发布章节这件事变得可靠、可查看、可暂停、可恢复。第一版暂时不做桌面软件，等核心流程稳定后再考虑封装成桌面版。

## 用户流程

1. 用户打开本地网页工具。
2. 用户选择一本小说的本地文件夹。
3. 工具使用文件夹名作为番茄后台书名。
4. 工具扫描文件夹里的章节文件。
5. 用户选择发布章节范围，例如第 5 章到第 80 章。
6. 工具生成发布计划。
7. 用户可以手动调整某一章的发布日期和发布时间。
8. 工具展示最终发布预览，让用户看到每一章会在哪一天、哪个时间发布。
9. 用户确认发布计划。
10. 工具启动一个独立可见的浏览器窗口。
11. 如果未登录，用户手动登录番茄作者后台。
12. 工具在番茄作者后台找到同名书籍。
13. 工具逐章提交，并开启定时发布。
14. 每章完成后，工具立即记录结果。
15. 如果遇到异常，工具立即暂停，等待用户处理后点击继续。

发布运行时，用户可以继续用电脑做其他事情，但不要操作工具正在控制的那个浏览器窗口。只有当工具暂停并提示需要人工处理时，用户再去那个浏览器窗口里处理登录、验证码、弹窗等问题。

## 章节导入规则

第一版支持“一章一个 Markdown 文件”。

支持的文件名格式：

```text
第001章 章节名.md
```

解析规则：

- 从文件名中解析章节序号。
- 从文件名中解析章节标题。
- 按章节序号排序。
- 每个 Markdown 文件第一行预计会重复章节标题，例如 `# 第001章 章节名`。
- 发布到番茄时，工具会自动去掉第一行标题，只把剩余内容作为章节正文。
- 工具会统计正文字符数，用于界面展示和发布前检查。

第一版不支持：

- 混乱或不统一的文件名格式。
- 一本书全部章节写在一个 Markdown 文件里。
- 自动修复章节标题。
- 自动改写、润色、校对正文。

## 发布排期

用户选择：

- 起始章节。
- 结束章节。
- 开始日期。

开始日期默认填今天，但用户可以在界面里修改。

默认每天发布时间：

- `09:30`：1 章
- `12:00`：1 章
- `19:00`：1 章
- `19:05`：1 章

用户可以在页面上维护每日发布时间点。时间点有几个，工具就按每天几章生成排期。比如保留 3 个时间点就是每天 3 章，新增到 5 个时间点就是每天 5 章。

工具按章节顺序生成排期。开始发布前，用户可以手动修改任意章节的日期或时间。

如果某一章已经记录为“已定时”或“已发布”，重新生成计划时会自动跳过，避免重复提交。

## 界面设计

第一版是本地网页工具，不是公开网站。

界面分为几个主要区域。

### 导入区

用户选择小说文件夹后，工具显示：

- 书名，也就是文件夹名。
- 找到的 Markdown 文件总数。
- 成功识别的章节数量。
- 是否读取到已有的 `.fanqie-publish.json` 发布记录。

### 章节范围和开始日期

用户填写：

- 起始章节号。
- 结束章节号。
- 开始日期，默认是今天。
- 每日发布时间点，默认是 `09:30`、`12:00`、`19:00`、`19:05`，可以新增、删除或修改。

工具会自动排除已经记录为“已定时”或“已发布”的章节。

### 发布计划区

工具以表格展示计划，每行一章：

- 章节号。
- 章节标题。
- 字符数。
- 计划日期。
- 计划时间。
- 当前状态。
- 失败原因。

开始发布前，日期和时间可以手动编辑。

用户点击开始发布前，工具必须展示最终确认预览。预览按日期和时间排序，清楚列出每一章会在具体哪一天、哪个时间发布。用户确认无误后，工具才允许开始自动提交。

### 发布控制区

按钮包括：

- 检查登录。
- 生成计划。
- 开始发布。
- 暂停。
- 继续。
- 停止。

## 数据存储

工具保存两份发布数据：

1. 工具内部数据：用于恢复界面和当前任务状态。
2. 小说文件夹里的 `.fanqie-publish.json`：用于保存这本书自己的发布记录。

这样即使以后换电脑、重装工具，或者重新导入同一个小说文件夹，也能读取这本书已有的发布记录，避免重复发布。

每章记录包含：

- 章节号。
- 章节标题。
- Markdown 文件名。
- 字符数。
- 计划发布日期。
- 计划发布时间。
- 实际提交时间。
- 状态。
- 失败原因。

支持的状态：

- `pending`：待发布。
- `scheduled`：已提交到番茄定时发布。
- `failed`：发布过程中失败并暂停。
- `skipped`：生成计划时被跳过。

## 浏览器自动化

第一版使用 Playwright，并使用独立浏览器环境。

行为规则：

- 浏览器窗口可见。
- 第一次运行可能需要用户手动登录。
- 登录状态尽量保存，后续运行复用。
- 默认不接管用户正在使用的 Chrome。
- 如果独立浏览器方案后续发现不稳定，再考虑支持使用用户已有的 Chrome 登录状态。

## 单章发布流程

每一章作为一个独立任务处理。成功或失败后，都立即写入记录。

单章流程：

1. 打开番茄作者后台。
   - 后台入口地址：`https://fanqienovel.com/main/writer/book-manage`
2. 找到与本地文件夹同名的书。
3. 进入章节发布页面。
4. 填写章节标题。
5. 填写章节正文。
6. 等待编辑器保存状态安全，或页面允许继续下一步。
7. 点击下一步。
8. 打开发布设置。
9. 选择“不使用 AI”，或者保持非 AI 发布状态。
10. 开启定时发布。
11. 填写发布日期和时间。
12. 点击确认发布。
13. 识别发布成功提示。
14. 将该章标记为 `scheduled`。
15. 继续下一章。

## 异常处理和恢复

工具遇到风险或不明确状态时，立即暂停。它不会跳过当前失败章节，也不会自动继续后面的章节。

暂停情况包括：

- 未登录。
- 出现验证码或验证要求。
- 找不到目标书籍。
- 找到多个同名或疑似重复书籍，无法判断。
- 无法打开章节发布页面。
- 找不到标题输入框、正文编辑器、下一步按钮、定时发布控件或确认发布按钮。
- 番茄提示字数限制、审核风险、发布失败等阻塞信息。
- 页面保存或跳转超时。
- 网络异常或浏览器自动化异常。

暂停后：

- 当前章节标记为 `failed`。
- 记录失败原因。
- 用户根据提示在浏览器里处理问题。
- 用户回到工具点击继续。
- 工具重新尝试当前章节。

已经记录为 `scheduled` 的章节，默认不会再次提交。

## 第一版范围

第一版包含：

- 一次只处理一本书。
- 一次只处理一个发布计划。
- 导入本地 Markdown 文件夹。
- 按文件名解析章节。
- 选择起始章节和结束章节。
- 生成可编辑的发布计划。
- 使用独立 Playwright 浏览器自动操作。
- 按文件夹名自动查找番茄后台书籍。
- 自动提交定时发布。
- 遇到异常立即暂停。
- 保存工具内部状态和 `.fanqie-publish.json` 发布记录。

第一版不包含：

- 多本书同时发布。
- 每本书不同的发布时间规则。
- 桌面软件安装包。
- 云端同步。
- 多账号管理。
- 自动处理验证码。
- 自动改写或校对正文。
- 当前章失败后自动跳过并继续后面的章节。
- 长期后台定时器。工具不是等到某个时间点才发布，而是提前把章节提交到番茄的定时发布系统。
- 绕过番茄平台限制或风控的能力。

## 推荐实现方式

第一版使用本地网页应用：

- 前端：本地仪表盘，用于导入、生成计划、编辑计划和控制发布。
- 后端：本地 API，用于读取文件、生成排期、保存记录和控制自动化。
- 自动化：Playwright，使用持久化浏览器配置保存登录状态。

这样核心流程更容易测试和调试。如果后续验证稳定，可以把同一套核心逻辑封装成桌面软件。

## 风险点

- 番茄作者后台页面结构可能变化，需要更新选择器。
- 正文编辑器可能是富文本编辑器，不能只按普通输入框处理。
- 定时发布日期和时间控件可能是自定义控件，需要专门适配。
- 登录、验证码、风控提示需要人工处理。
- 点击确认发布后的成功信号，需要在真实页面上验证。

## 验收标准

- 用户可以导入包含 `第001章 章节名.md` 这类文件的小说文件夹。
- 工具可以提取章节号、标题、正文和字符数。
- 用户可以选择章节范围和开始日期。
- 工具默认可以按 `09:30`、`12:00`、`19:00`、`19:05` 生成每天四章的排期。
- 用户可以在页面上新增、删除或修改每日发布时间点，并用时间点数量决定每天发布几章。
- 用户可以在发布前编辑计划日期和时间。
- 工具可以打开独立浏览器，并尽量保存登录状态。
- 工具可以按文件夹名找到番茄后台对应书籍。
- 工具可以逐章提交定时发布。
- 每章成功后记录为 `scheduled`。
- 重新生成计划时，工具会跳过已经 `scheduled` 的章节。
- 遇到阻塞错误时，工具暂停，并在用户处理后从当前章节继续。
